TunnelEdge

A self-hosted reverse-tunnel platform built on QUIC. Expose any local TCP service to the internet without opening firewall ports — via encrypted QUIC multiplexed streams and SNI-based domain routing.

Quick Start

The fastest way to get TunnelEdge running is with Docker Compose. The full stack — registry, gateway, dashboard, and Mailpit for email — starts with one command.

Prerequisites

  • Docker + Docker Compose v2
  • A domain with a wildcard DNS record pointing to your server (e.g. *.tunneledge.example.com → 1.2.3.4)
  • Go 1.22+ (only required if building from source)

1. Clone and start

git clone https://github.com/uloydev/tunneledge
cd tunneledge
docker compose -f deployments/docker/docker-compose.yml up -d

2. Register an account

Open http://localhost:8082, create an account, and verify your email via Mailpit at http://localhost:18025.

3. Create an agent

In the dashboard go to Agents → New agent. Copy the token shown — it is displayed only once.

4. Configure and run the agent

# config/agent.yaml
agent:
  gateway_addr: your-server.example.com:4433
  token:        te_xxxxxxxxxxxxxxxx
  tunnels:
    - label:      web
      local_addr: localhost:3000
./bin/agent -c config/agent.yaml

Your service is now live at https://web-<agent-id>.yourdomain.example.com.

Docker Deploy

The deployments/docker/docker-compose.yml file defines the full production stack.

Services

  • postgres — PostgreSQL 16 (persistent volume pgdata)
  • registry — gRPC session registry on port 50051
  • gateway — QUIC agent listener :4433/udp, public TLS :443/tcp, metrics :9090
  • dashboard — HTTP API + HTMX web UI on :8080 (mapped to 8082 in dev)
  • mailpit — Dev SMTP server; web UI on :18025
  • http-echo — Sample echo service for testing tunnels

TLS certificates

Generate self-signed wildcard certs for local development:

make certs

For production use a real wildcard certificate from Let's Encrypt or your CA. Mount certs/ into the gateway container and set TE_GATEWAY_TLS_CERT_FILE / TE_GATEWAY_TLS_KEY_FILE.

Architecture

TunnelEdge is four cooperating services. Each is a standalone Go binary.

┌────────────────┐
│   CLI Agent    │   ./bin/agent
│   Go + QUIC    │
└───────┬────────┘
        │ QUIC/TLS 1.3  (port 4433)
┌───────▼────────┐        Public clients
│   Gateway      │◄─────── TLS SNI (port 443)
└───────┬────────┘
        │ gRPC  (port 50051)
┌───────▼────────┐
│   Registry     │   session & routing state
└────────────────┘
        │ HTTP REST + SSE  (port 8080)
┌───────▼────────┐
│   Dashboard    │   agents · tunnels · sessions
└────────────────┘

Data flow (incoming public request)

  • Public client connects to web-myserver.yourdomain.dev:443 via TLS
  • Gateway reads the TLS SNI hostname during the handshake
  • TunnelRouter maps hostname → tunnel ID (in-memory, populated by registry)
  • Gateway opens a new QUIC stream to the agent holding that tunnel
  • Agent accepts the stream, dials the local TCP service
  • Bidirectional relay begins — data flows transparently at wire speed

Stream multiplexing

Each agent holds a single QUIC connection to the gateway. Incoming public TCP connections each become independent QUIC streams on that connection — no head-of-line blocking, no per-connection TLS overhead.

Wire Protocol

The agent↔gateway control channel runs over a dedicated QUIC stream. Every frame starts with a 1-byte message type tag.

Byte  Name              Direction    Version
0x00  MsgHello          agent→gw     v2+
0x01  MsgAuth           agent→gw     v1 (legacy)
0x02  MsgAuthResponse   gw→agent     v1
0x03  MsgData           both         v1+
0x04  MsgHeartbeat      agent→gw     v1+
0x05  MsgClose          both         v1+
0x06  MsgAuthV2         agent→gw     v2+
0x07  MsgAuthRespV2     gw→agent     v2+
0x08  MsgStreamLabel    gw→agent     v2+

Connection handshake (v2)

  • Agent opens QUIC control stream, sends MsgHello: [0x00][version uint16][len uint16][clientVersion string]
  • Current ProtocolVersion = 2. Gateway closes the connection if versions differ.
  • Agent immediately follows with MsgAuthV2: token + list of {label, local_addr} tunnel entries
  • Gateway validates the token (HMAC pre-filter → bcrypt), registers tunnels, replies with MsgAuthRespV2

Note: MsgAuth / MsgAuthResponse (v1) are still parsed for backward compatibility but v2 clients always send MsgHello first.

Authentication

Agent tokens

Tokens are random 32-byte hex strings (te_<hex>). The gateway stores only the bcrypt hash — the raw token is shown once in the dashboard at creation time.

On each QUIC connection the gateway authenticates as follows:

  • If an HMAC fingerprint is stored alongside the hash (set at token-issuance time), the gateway computes a constant-time HMAC comparison first, skipping bcrypt for non-matching entries
  • Only the single matching entry incurs the ~100 ms bcrypt cost
  • Rotate tokens any time via the dashboard or POST /api/v1/agents/:id/rotate-token

Dashboard users

Passwords are bcrypt-hashed at registration. Sessions use short-lived JWTs (HS256, 24 h default). Email verification is required before login. Rate limiting applies to /api/v1/auth/login and /api/v1/auth/register (5 requests/min per IP).

Config Files

Each service reads a YAML config file from config/<service>.yaml (or the path passed with -c). All keys are nested under their service section. Example:

# config/gateway.yaml
log:
  level:  info
  format: json

gateway:
  quic_listen_addr:   :4433
  public_listen_addr: :443
  base_domain:        tunneledge.example.com
  registry_addr:      localhost:50051
  tls_cert_file:      certs/server.crt
  tls_key_file:       certs/server.key

db:
  driver: postgres
  dsn:    host=localhost port=5432 user=te password=te dbname=te sslmode=disable
# config/agent.yaml
agent:
  gateway_addr:        gateway.example.com:4433
  token:               te_xxxxxxxxxxxxxxxx
  reconnect_delay:     2s
  max_reconnect:       0          # 0 = unlimited
  heartbeat_interval:  15s
  tls_insecure:        false
  tunnels:
    - label:      web
      local_addr: localhost:3000
    - label:      api
      local_addr: localhost:8080

REST API — Auth

Base URL: http://your-dashboard:8080/api/v1. All protected endpoints require Authorization: Bearer <jwt> (or a session HTTP-only cookie set by login).

POST /auth/register

Rate-limited to 5 req/min per IP.

// Request
{ "email": "user@example.com", "password": "...", "name": "Alice" }

// 201 Created
{ "message": "registration successful, please check your email" }

POST /auth/login

Rate-limited to 5 req/min per IP. Returns a JWT and sets a session cookie.

// Request
{ "email": "user@example.com", "password": "..." }

// 200 OK
{ "token": "eyJ...", "expires_at": 1747000000 }

POST /auth/logout

Clears the session cookie. No body required.

POST /auth/resend-verification

Body: {"email": "..."}. Always returns 200 (no enumeration).

GET /auth/me 🔒

Returns the authenticated user's profile.

// 200 OK
{ "id": 1, "email": "user@example.com", "name": "Alice", "created_at": "..." }

REST API — Agents 🔒

Method + PathDescription
POST /agentsCreate agent. Returns token (shown once). Body: {"name":"…","agent_id":"…"}
GET /agentsList all agents owned by the authenticated user
GET /agents/:idGet a single agent by ID
PUT /agents/:idUpdate agent name. Body: {"name":"…"}
DELETE /agents/:idDelete agent and all its tunnels
POST /agents/:id/rotate-tokenRotate the agent's auth token. Returns new raw token once.

Agent response shape

{ "id": 1, "name": "my-server", "agent_id": "my-server",
  "created_at": "2026-05-15T…", "updated_at": "2026-05-15T…" }

REST API — Tunnels 🔒

Tunnels are nested under agents: /agents/:id/tunnels.

Method + PathDescription
POST /agents/:id/tunnelsCreate tunnel. Body: {"label":"web","local_addr":"localhost:3000"}
GET /agents/:id/tunnelsList all tunnels for the agent
GET /agents/:id/tunnels/:tidGet a single tunnel
PUT /agents/:id/tunnels/:tidUpdate label or local address
DELETE /agents/:id/tunnels/:tidDelete a tunnel

REST API — Sessions 🔒

Active sessions are live tunnel connections registered by the gateway.

Method + PathDescription
GET /sessionsList all active sessions
DELETE /sessions/:tunnelIDForce-close a session
GET /agents/:id/statusGet the active session for a specific agent
GET /eventsSSE stream — real-time session and agent state changes

Session response shape

{ "tunnel_id": "t-my-server", "agent_id": "my-server",
  "public_addr": "web-my-server.example.com:443",
  "local_addr":  "localhost:3000",
  "created_at":  1747000000,
  "last_heartbeat": 1747000120 }
